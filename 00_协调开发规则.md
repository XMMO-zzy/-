## 2. 三大冻结接口（V1.0）

| 接口 | Owner | 冻结内容 |
|------|-------|----------|
| 数据写库接口 | 外部硬件后端 + 团队成员3 | device_id / node_id / timestamp / fault_class / confidence / health_score / vib_rms / temperature / current / need_deep_confirm（字段与类型冻结） |
| 告警规则接口 | 团队成员2 + 团队成员3 | 一级：温度>80°C或电流>120%额定；二级：fault_class!=normal且confidence>80；低置信度上送：confidence<70 |
| 展示API接口 | 团队成员3 + 团队成员4 | 设备状态枚举、告警状态流转、分页与查询参数 |

接口冻结后，禁止单方面修改。任何修改需走第4节变更流程。

## 3. V1.0阈值冻结表

```
温度上限         : 80 °C
电流上限         : 120% 额定
高置信度告警     : confidence > 80 且 fault_class != normal
低置信度分流     : confidence < 70 自动进入深度分析
分类标签枚举     : normal | bearing_fault | winding_fault | rotor_fault | overload
帧头             : 0xAA55
单帧大小         : 40 ~ 80 字节
上报频率         : 1 ~ 10 Hz
```

## 4. 变更管理流程

1. 提出：在问题清单中提交变更单（标注影响范围、原因、风险）
2. 评审：周会评审，相关Owner必须到场
3. 实现：相关角色同步开发、互相通知
4. 回归：团队成员5执行回归测试，覆盖受影响用例
5. 升版：版本号统一升级（如`V1.0 → V1.1`），更新日志归档

无评审记录的变更不允许进入主分支。
