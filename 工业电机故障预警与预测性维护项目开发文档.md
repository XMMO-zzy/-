# AI工业电机故障预警与预测性维护项目开发文档

## 1. 项目概述

> 范围声明（当前执行口径）：硬件采集与其写库后端由外部团队负责；本项目仅负责 Web 前后端与数据库直连能力（读库、业务处理、可视化、告警与报表）。

### 1.1 项目背景

传统电机监测方案依赖有线接入识别数据来源，部署不灵活、布线成本高，且所有计算集中在上位机端。本项目引入**远端计算**概念——在采集端（传感器+MCU）完成数据预处理、边缘AI推理和智能标注，通过WiFi/BLE等无线方式将标注后的结构化数据传递到后端，实现"即收即识别"的智能数据流。

### 1.2 项目目标

| 目标 | 说明 |
|------|------|
| 实时监测 | 基于数据库中的遥测数据实现实时监测 |
| 数据接入 | 对接外部硬件后端写库结果，保障字段一致与可追溯 |
| 传输范围 | 本项目不包含采集侧无线传输实现（由外部团队负责） |
| 故障识别 | 边缘AI + 后端深度模型 + 大模型三层架构 |
| 预测性维护 | 故障分类、劣化评估、剩余寿命预测 |
| 大模型集成 | 自建数据库存储标注数据，调用通用大模型深度分析 |

### 1.3 与传统方案对比

| 维度 | 传统方案 | 本方案 |
|------|----------|--------|
| 数据来源识别 | 依赖有线端口号 | 数据包自带设备ID，无线即可识别 |
| 传输数据量 | 原始数据全量传输 | 仅传标注摘要，带宽节省>98% |
| 计算负载 | 上位机承担全部 | 边缘节点分担预处理和AI推理 |
| 响应延迟 | 依赖上位机 | 边缘毫秒级本地预判 |
| 部署方式 | 有线布线，位置固定 | 无线部署，灵活扩展 |

## 2. 系统架构

### 2.1 分层架构

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  传感器采集      │────>│  远端计算节点    │────>│  无线传输层      │
│  (振动/温度/电流) │     │  (预处理+标注+AI) │     │  (WiFi/BLE/LoRa) │
└──────────────────┘     └──────────────────┘     └──────────────────┘
                                                          │
                              ┌────────────────────────────┘
                              ↓
                    ┌──────────────────┐     ┌──────────────────┐
                    │  后端接收与存储  │────>│  AI深度分析层    │
                    │  (自建数据库)    │     │  (深度模型+大模型)│
                    └──────────────────┘     └──────────────────┘
                              │                       │
                              ↓                       ↓
                    ┌──────────────────┐     ┌──────────────────┐
                    │  可视化上位机    │     │  预警运维应用    │
                    └──────────────────┘     └──────────────────┘
```

### 2.2 三层AI架构

| 层级 | 运行位置 | 模型 | 功能 | 延迟 |
|------|----------|------|------|------|
| 边缘AI | 远端节点(MCU) | 轻量CNN/决策树(INT8) | 实时分类、健康评分 | <10ms |
| 深度分析 | 上位机 | 1D-CNN + LSTM | 精细分类、寿命预测 | <100ms |
| 大模型 | 云端API | Claude/GPT/DeepSeek | 故障挖掘、关联分析 | 秒级 |

### 2.3 核心数据流

```
外部硬件后端写库 → 本项目后端直连读取 → 数据校验与业务路由
    → 深度模型精细分析 → 大模型深度挖掘 → 预警/维护建议
    → Web看板/告警中心/设备页/报表
```

## 3. 远端计算节点设计

### 3.1 硬件选型方案

**方案A：现有模块方案（推荐）** — ESP8266 + HC-05，WiFi主通道+蓝牙调试通道，~$4-10/节点

| 组件 | 型号 | 说明 |
|------|------|------|
| 主控 | ESP8266(如NodeMCU) | 集成2.4G WiFi，成本低，适合数据采集与传输 |
| 蓝牙模块 | HC-05 | 串口透传蓝牙2.0，用于近距离调试与应急数据回传 |
| 振动 | ISM330DHCX | 6轴IMU，内置ML Core硬件分类引擎 |
| 温度 | DS18B20 | 数字温度传感器 |
| 电流 | ACS712 | 霍尔电流传感器 |

### 3.2 传感器采集参数

| 传感器 | 采样频率 | 说明 |
|--------|----------|------|
| ISM330DHCX | 100-400Hz | 振动特征采集，ML Core硬件预分类 |
| IIS3DWB（可选） | 最高6.3kHz | 宽带工业振动监测 |
| DS18B20 | 1Hz | 温度趋势监控 |
| ACS712 | 10Hz | 电流异常和谐波监测 |

### 3.3 边缘AI框架

| 框架 | 适用主控 | 特点 |
|------|----------|------|
| TFLite Micro | ESP8266(轻量特征规则)/ESP32-S3/STM32 | 跨平台，INT8量化，推荐生产部署 |
| Edge Impulse | ESP32-S3/STM32 | 端到端ML流水线，推荐快速原型（ESP8266侧可做特征阈值版） |
| STM32Cube.AI | STM32系列 | CMSIS-NN加速，含预测性维护参考固件 |

### 3.4 节点软件流程

```
采集 → 中值滤波去野值 → 移动平均平滑 → 256点FFT频谱分析
    → 时域特征(RMS/峰值/峭度/波峰因子) → 数据质量评估
    → INT8量化CNN推理 → 输出{分类标签, 置信度, 健康评分}
    → 组装标注数据包 → WiFi/MQTT发送
```

分类标签枚举：`normal | bearing_fault | winding_fault | rotor_fault | overload`

模型规格：ESP8266方案建议采用规则+轻量模型（<80KB）；若升级ESP32-S3可支持50-200KB INT8模型与更低推理延迟

## 4. 数据标注协议

标注协议是远端节点与后端之间的核心契约。考虑ESP8266带宽和内存开销，链路层采用**二进制帧**发送，后端统一解包为结构化数据（调试模式可保留JSON透传）：

| 字段 | 字节数 | 说明 |
|------|--------|------|
| Header | 2 | 帧头 `0xAA55` |
| Version | 1 | 协议版本 |
| MsgType | 1 | `0x01`遥测，`0x02`告警 |
| NodeId | 2 | 节点ID(无符号整型) |
| DeviceId | 2 | 电机ID(无符号整型) |
| Timestamp | 4 | Unix时间戳(s) |
| FaultClass | 1 | 故障分类枚举 |
| Confidence | 1 | 置信度(0-100) |
| HealthScore | 1 | 健康分(0-100) |
| VibRms | 2 | 振动RMS，放大1000倍存储 |
| Temp | 2 | 温度，放大100倍存储 |
| Current | 2 | 电流，放大100倍存储 |
| CRC16 | 2 | 校验码(Modbus CRC16) |

**关键字段说明**：

| 字段 | 作用 |
|------|------|
| NodeId / DeviceId | 后端自动识别来源，无需有线端口 |
| FaultClass | 即收即知故障类型，可直接触发预警 |
| Confidence | 低置信度(<70)标记为需后端二次确认 |
| HealthScore | 0-100健康评分，量化设备状态 |
| VibRms / Temp / Current | 核心特征摘要，供趋势分析和大模型训练 |
| CRC16 | 传输完整性校验，降低误码导致的误判风险 |

## 5. 数据传输通信层

### 5.1 通信方式

| 方式 | 带宽 | 距离 | 用途 | 实现 |
|------|------|------|------|------|
| WiFi(ESP8266) | Mbps级 | 30-80m | 厂内主数据通道(MQTT/TCP) | ESP8266模块 |
| 蓝牙(HC-05) | ~100kbps | 5-10m | 配置/调试/近场应急透传 | HC-05串口模块 |
| LoRa | kbps级 | 2-15km | 跨厂区远距离备选 | SX1262模块 |
| 串口USB | 高 | 有线 | 实验室调试 | MCU USART |

### 5.2 外部数据接入方案（主通道）

本项目默认通过数据库直连接入数据：

- 外部硬件后端按约定字段写入 `annotated_data` / `alerts`
- 本项目后端按游标或时间窗口拉取新增记录
- 对异常字段进行隔离并记录日志，不阻塞主链路
- 对重复数据执行幂等治理（以 `device_id + timestamp` 为主键语义）

### 5.3 传输优化

| 数据类型 | 单帧大小 | 频率 | 带宽 |
|----------|----------|------|------|
| 原始数据（传统） | ~2KB | 100Hz | ~200KB/s |
| 标注摘要（二进制） | ~40-80B | 1-10Hz | ~0.04-0.8KB/s |

智能策略：正常时每秒1次摘要，异常时立即发详细特征+触发原始数据上传。

## 6. AI算法层

### 6.1 数据集构建

- 公开工业电机故障标准数据集 + 实验室模拟采集 + 现场运行数据
- 边缘标注数据可反哺训练集
- 预处理：归一化[0,1]、时域特征(RMS/峰值/峭度)、频域特征(FFT)、数据增强

### 6.2 故障分类模型

- 结构：3层1D-CNN → MaxPooling → 2层全连接 → Softmax(5类)
- 参数：卷积核3，滤波器32/64/128，dropout 0.3
- 边缘版：INT8量化，<200KB，裁剪为3类(正常/异常/严重)

### 6.3 寿命预测模型

- 结构：LSTM → 全连接 → 回归输出(健康度+剩余寿命)
- 输入：标注数据特征值时间序列

### 6.4 性能指标

| 指标 | 目标 |
|------|------|
| 故障分类准确率（后端） | >95% |
| 边缘分类准确率 | >90%（量化损失<5%） |
| 寿命预测误差 | <10% |
| 边缘推理时间 | <10ms |

## 7. 数据库设计

### 7.1 双库架构

本项目采用**关系型数据库 + 向量数据库**双库架构：

```
标注数据 ──→ 关系型数据库(SQLite/MySQL)    ──→ SQL查询、统计报表、预警记录
    │
    └────→ 向量数据库(Milvus/ChromaDB)     ──→ 相似故障检索、大模型RAG、异常模式匹配
```

| 数据库类型 | 产品 | 存储内容 | 用途 |
|------------|------|----------|------|
| 关系型 | SQLite(单机) / MySQL(生产) | 标注数据、预警、维护记录、大模型分析记录 | 结构化查询、统计聚合、业务逻辑 |
| 向量数据库 | Milvus(生产) / ChromaDB(轻量) | 传感器特征向量 + 故障标签embedding | 相似故障检索、大模型RAG上下文、异常模式匹配 |
| 时序数据库（可选） | TDengine / InfluxDB | 高频时序数据 | 大规模时序存储和压缩 |

### 7.3 向量数据库设计

#### 7.3.1 选型

| 产品 | 部署方式 | 特点 | 适用场景 |
|------|----------|------|----------|
| ChromaDB | 嵌入式(pip install) | 零配置，Python原生 | 单机/实验室/原型 |
| Milvus | Docker/K8s | 高性能，支持亿级向量，分布式 | 生产部署 |
| Qdrant | Docker | Rust实现，REST API | 中等规模生产 |
| pgvector | PostgreSQL插件 | 与关系型统一，运维简单 | 已用PostgreSQL的场景 |

推荐：原型用ChromaDB，生产用Milvus或pgvector。

#### 7.3.2 向量Collection设计

**fault_vectors** — 故障特征向量集合

| 字段 | 类型 | 说明 |
|------|------|------|
| vector_id | STRING | 唯一ID，关联annotated_data.vector_id |
| embedding | FLOAT_VECTOR(32) | 传感器特征向量（归一化） |
| device_id | STRING | 设备标识（元数据过滤） |
| edge_class | STRING | 故障分类标签 |
| health_score | FLOAT | 健康评分 |
| timestamp | INT64 | 时间戳 |

32维向量构成：振动特征(8维) + 温度特征(2维) + 电流特征(2维) + FFT频带能量(20维)

#### 7.3.3 向量入库流程

```
标注数据到达 → 提取特征值 → 归一化为32维向量
    ├──→ 写入向量数据库(fault_vectors)，获得vector_id
    └──→ 写入关系型数据库(annotated_data)，记录vector_id关联
```

#### 7.3.4 核心应用场景

| 场景 | 输入 | 输出 | 价值 |
|------|------|------|------|
| 相似故障检索 | 当前异常特征向量 | 历史最相似的N条故障记录 | 快速定位同类故障和处理方案 |
| 大模型RAG上下文 | 分析请求的特征向量 | 最相关历史案例作为Prompt上下文 | 大模型分析更精准，减少幻觉 |
| 异常模式匹配 | 新数据特征向量 | 与已知故障模式的匹配度 | 发现新型故障模式 |
| 跨设备相似性 | 某设备故障向量 | 其他设备中相似特征数据 | 发现跨设备共性问题 |

### 7.4 ER关系

```
devices 1──N edge_nodes
devices 1──N annotated_data ←──vector_id──→ fault_vectors(向量库)
devices 1──N alerts
devices 1──N maintenance_records
edge_nodes 1──N annotated_data
alerts N──1 maintenance_records (可选)
llm_analysis ──rag_context_ids──→ fault_vectors(向量库)
```

## 8. 大模型集成

### 8.1 核心链路

```
自建数据库 → SQL查询聚合 → 构造特征向量 → 向量数据库检索相似案例(RAG)
    → 聚合数据+相似案例 → 构造Prompt → 调用大模型API → 解析结果
    → 存回关系型数据库(llm_analysis) → 展示/执行
```

### 8.2 模型选型

| 大模型 | 特点 | 适用场景 | 费用 |
|--------|------|----------|------|
| Claude Sonnet | 长上下文200K，结构化输出好 | 大批量分析、报告 | $3/$15 per MTok |
| GPT-4o | 多模态，生态成熟 | 通用分析 | $2.5/$10 per MTok |
| DeepSeek-V3 | 性价比高，中文强 | 日常分析 | ¥1/¥2 per MTok |
| 本地Qwen/Llama | 数据不出厂 | 数据敏感/离线 | 仅硬件成本 |

策略：日常用DeepSeek，关键分析用Claude，敏感场景用本地模型。

### 8.3 分析场景

| 场景 | 数据库输入 | 输出 | 触发 |
|------|------------|------|------|
| 故障模式挖掘 | 多设备30天标注统计 | 隐性故障模式 | 每周定时 |
| 跨设备关联 | 同产线异常数据 | 故障传导关系 | 多设备同时异常 |
| 维护策略优化 | 故障+维护+运行记录 | 最优维护计划 | 每月定时 |
| 异常报告 | 单次异常完整标注 | 自然语言报告 | 三级预警触发 |
| 健康趋势 | 健康评分时间序列 | 劣化预测 | 评分持续下降 |

### 8.4 Prompt设计要点

- 角色设定：工业电机故障诊断专家
- 输入结构：设备信息 + 标注数据统计摘要 + 异常详情
- 输出要求：JSON格式，含故障判断、严重程度、根因、维护建议、是否停机
- 模板集中管理，版本化

### 8.5 降级与安全

| 策略 | 说明 |
|------|------|
| API降级 | 失败时静默降级，不影响核心预警 |
| 多模型切换 | Claude → DeepSeek → 本地模型自动切换 |
| 数据脱敏 | 去除位置/人员信息，仅保留编号和数值 |
| 费用控制 | 每日Token上限，超限切换低成本模型 |
| 结果审核 | 维护建议需人工确认后执行 |
| 缓存 | 相似查询缓存7天 |

### 9.3 数据接收逻辑

- 数据来源：外部硬件后端写入数据库
- 本项目后端读取后统一处理：字段校验+业务路由 → 更新界面
- 低置信度数据(confidence < 70)自动提交深度模型二次确认
- 边缘AI已识别异常(confidence > 0.8)直接触发预警

### 9.4 后端处理流水线（实施版）

```
数据库新增数据读取
  → 字段校验与规范化
  → 入库治理(去重/隔离)
  → 告警分级路由
  → 推送前端实时通道
```

实施约束：
- 字段校验失败写入错误日志队列，不阻塞主消费线程
- 同一 `device_id + timestamp` 采用幂等写入，避免重传重复数据
- 低置信度（confidence<70）自动打标 `need_deep_confirm=1`
- 告警消息需保留 `source(edge/deep/llm)` 供前端分层展示

### 9.5 前端告警闭环（实施版）

- 实时看板：展示温度/电流/振动与健康分，刷新间隔建议1s
- 告警中心：按级别、来源、设备筛选；支持确认、关闭、追溯
- 设备页：显示最新状态、最近24h告警、维护建议
- 报表导出：支持按日/周导出告警与健康评分趋势

## 10. 预警与预测性维护

### 10.1 四级预警机制

| 级别 | 来源 | 触发条件 | 处理 |
|------|------|----------|------|
| 一级（即时） | 远端节点 | 参数超限(温度>80°C, 电流>120%额定) | 本地声光报警 + MQTT推送 |
| 二级（边缘AI） | 远端节点 | 边缘AI识别故障(confidence>0.8) | MQTT推送 + 上位机弹窗 |
| 三级（深度分析） | 上位机 | 深度模型确认严重风险 | 强制断电 + 维护工单 |
| 四级（趋势） | 大模型 | 健康评分持续下降 | 预测性维护计划 |

### 10.2 维护建议

| 故障类型 | 建议 |
|----------|------|
| 轴承异常 | 润滑保养，检查磨损，计划更换 |
| 绕组故障 | 检查绝缘，降负载，必要时更换 |
| 转子故障 | 检查平衡，校正偏心 |
| 过载过热 | 检查负载，优化参数，加强通风 |
| 电流异常 | 排查缺相/短路，检查供电质量 |
